# 用命令行工具直接修改 `minimal` 二进制的退出码

这篇文档总结一次非常典型的“二进制补丁”过程：

- 不改源代码
- 不重新编译
- 只用命令行工具
- 直接把现有 ELF 可执行文件 `minimal` 的退出码改成 `42`

目标文件：

- `OS_Spring_2026/course_2/minimal/minimal`

## 1. 这次修改的目标是什么

`minimal` 这个程序在 `x86_64` 分支里，最终会执行一条退出系统调用：

```asm
mov $SYS_exit, %rax
mov $812, %rdi
syscall
```

其中：

- `rax` 里放系统调用号 `SYS_exit`
- `rdi` 里放退出码

所以只要把“装进 `rdi` 的立即数”从 `812` 改成 `42`，运行结果就会变。

## 2. 第一步：先反汇编，确认机器码里真的有这条指令

命令：

```bash
objdump -d OS_Spring_2026/course_2/minimal/minimal | sed -n '1,80p'
```

关键输出：

```text
40101e: 48 c7 c0 3c 00 00 00  mov    $0x3c,%rax
401025: 48 c7 c7 2c 03 00 00  mov    $0x32c,%rdi
40102c: 0f 05                 syscall
```

解释：

- `0x32c` 就是十进制 `812`
- 说明当前二进制里，退出码确实已经被编进机器码了

## 3. 第二步：把虚拟地址和文件偏移对起来

我们不能直接对着虚拟地址 `0x401028` 写文件，必须先知道它在文件里的偏移。

命令：

```bash
readelf -l OS_Spring_2026/course_2/minimal/minimal
```

关键输出：

```text
LOAD           0x0000000000001000 0x0000000000401000 ...
```

这表示：

- 文件偏移 `0x1000`
- 会被加载到虚拟地址 `0x401000`

所以：

- 指令 `mov $0x32c,%rdi` 从虚拟地址 `0x401025` 开始
- 它的立即数从 `0x401028` 开始
- 对应的文件偏移就是：

```text
0x401028 - 0x401000 + 0x1000 = 0x1028
```

这就是我们真正要改的位置。

## 4. 第三步：用十六进制工具验证文件偏移附近的字节

命令：

```bash
xxd -g 1 -s 0x1018 -l 24 OS_Spring_2026/course_2/minimal/minimal
```

关键输出：

```text
00001018: 1c 00 00 00 0f 05 48 c7 c0 3c 00 00 00 48 c7 c7
00001028: 2c 03 00 00 0f 05 ...
```

解释：

- `48 c7 c7` 是 `mov imm32, %rdi`
- 后面的 `2c 03 00 00` 是小端序的 `0x0000032c`
- 也就是十进制 `812`

这一步的意义是：

> 在真正下手改之前，先确认“你准备改的字节确实就是目标字段”。

## 5. 第四步：直接打补丁

我们要把 `812` 改成 `42`。

十六进制里：

- `42` 是 `0x2a`
- 32 位小端序写法是：

```text
2a 00 00 00
```

命令：

```bash
printf '\x2a\x00\x00\x00' | dd of=OS_Spring_2026/course_2/minimal/minimal bs=1 seek=$((0x1028)) conv=notrunc
```

解释：

- `printf` 生成 4 个目标字节
- `dd` 把这 4 个字节写到文件偏移 `0x1028`
- `conv=notrunc` 表示不要把整个文件截断

## 6. 第五步：再反汇编验证一遍

命令：

```bash
objdump -d OS_Spring_2026/course_2/minimal/minimal | sed -n '1,30p'
```

关键输出：

```text
401025: 48 c7 c7 2a 00 00 00  mov    $0x2a,%rdi
```

说明补丁已经成功写进去了。

## 7. 第六步：真正运行，检查退出码

命令：

```bash
cd OS_Spring_2026/course_2/minimal
./minimal
echo $?
```

结果应当是：

```text
42
```

## 8. 这次 hack 的原理到底是什么

本质上你做的不是“魔法破解”，而是下面这几步：

1. 找到程序里对应逻辑的机器指令
2. 读懂这条指令的编码
3. 确定它在 ELF 文件中的字节位置
4. 把对应字段改成新的值

也就是说，这次修改利用的是一个非常朴素的事实：

> 可执行文件本质上就是一串字节。

而这些字节里，有些是指令，有些是数据；只要你知道规则，就能精确改写其中的一部分。

## 9. 为什么这次能改得这么简单

因为这次目标非常友好：

- 退出码是一个直接写进指令里的常量
- 指令长度没变
- 新值 `42` 也能放进同样大小的立即数字段

所以不需要：

- 重定位
- 改 section 大小
- 改 ELF 头
- 改跳转目标

这类修改属于最简单的一种二进制 patch：**原地覆盖常量**。

## 10. 这类修改的限制

这次方法虽然有效，但边界很明显。

它适合：

- 改常量
- 改某条指令的小参数
- 改一些不影响布局的短字段

它不适合直接拿来做：

- 大规模插入新逻辑
- 改变控制流长度很多的跳转
- 增减 section / segment
- 修改依赖重定位的大结构

## 11. 还要注意一件事

如果你之后重新执行：

```bash
make clean
make
```

那么 `minimal` 会按源代码 `minimal.S` 重新生成。

这意味着：

- 你刚才对二进制打的补丁会被覆盖
- 因为源码才是“重新构建时的真实来源”

所以这次 hack 是对当前二进制文件的修改，不是对源代码逻辑的永久修改。

## 12. 最后总结

这次过程可以压缩成一句话：

> 用 `objdump` 找指令，用 `readelf` 对地址和偏移，用 `xxd` 看字节，再用 `dd` 直接改文件。

这正好说明了一件很重要的事：

> 可执行文件不是神秘黑盒，它只是按一定格式组织起来的字节序列。
