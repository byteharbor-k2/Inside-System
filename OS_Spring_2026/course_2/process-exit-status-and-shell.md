# 进程退出、shell 状态码与信号终止

这篇文档总结三个经常混在一起的问题：

1. `exit()`、`return`、`_exit()` 的区别
2. shell 里的 `$?` 和 `waitpid()` 的关系
3. 为什么很多 shell 会把“被信号终止”显示成 `128 + signal`

## 1. 先抓住主线

一个程序结束时，并不是只做“退出”这一件事。

它至少涉及三层：

- 程序自己如何结束当前执行流
- C 运行时是否要做清理工作
- 父进程如何拿到这个程序的结束结果

所以很多看起来相似的概念，其实在不同层。

## 2. `return`、`exit()`、`_exit()` 的区别

### 2.1 `return`

`return` 最基本的作用是：

> 从当前函数返回给调用者。

例如：

```c
int add(int a, int b) {
    return a + b;
}
```

这里只有普通函数返回，没有“退出进程”这回事。

但如果 `return` 出现在 `main()` 里，含义会特殊一些：

```c
int main() {
    return 42;
}
```

这通常等价于：

```c
int main() {
    exit(42);
}
```

也就是说，在 `main()` 里，`return` 的效果通常是“正常结束整个进程，并把返回值交给运行时和操作系统”。

### 2.2 `exit(status)`

`exit()` 是 C 标准库提供的函数。

它做的不是“立刻消失”，而是：

1. 执行用户态的一些收尾工作
2. 然后再结束进程

典型收尾包括：

- 刷新 `stdio` 缓冲区
- 调用 `atexit()` 注册的函数
- 做运行时的一些清理

所以可以把 `exit()` 理解成：

> 先收拾现场，再走人。

### 2.3 `_exit(status)`

`_exit()` 更底层，也更“冷酷”。

它基本可以理解成：

> 不做 C 库层面的善后，直接结束当前进程。

也就是说，它通常不会：

- 刷新 `stdio` 缓冲
- 调用 `atexit()` 回调

所以它更接近“立刻从用户态回到内核，要求终止进程”。

### 2.4 一句话对比

- `return`：从函数返回
- `return` in `main`：正常结束程序
- `exit()`：带清理地结束程序
- `_exit()`：不清理，直接结束程序

## 3. 为什么 `_exit()` 存在

最常见的场景是 `fork()` 之后的子进程。

因为 `fork()` 会复制父进程的用户态状态，包括很多运行时缓冲区。

如果子进程在某些场景里调用 `exit()`，可能会把本来不想重复做的清理逻辑再做一遍，甚至把缓冲区重复刷新。

这时候往往更安全的做法是：

- 子进程要么立刻 `exec`
- 要么失败后直接 `_exit()`

所以 `_exit()` 存在的意义，不是“它更高级”，而是“它更原始、更少副作用”。

## 4. shell 里的 `$?` 是什么

在 shell 里：

```bash
echo $?
```

表示的是：

> 上一条命令的退出状态。

但这里有个很重要的细节：

- `$?` 不是程序自己直接塞给 shell 的一个任意整数
- shell 实际上是先通过 `wait()` / `waitpid()` 拿到子进程状态
- 然后再把这个状态整理成一个你熟悉的数字

所以 `$?` 是 shell 暴露给用户的“简化版结果”。

## 5. `waitpid()` 到底拿到了什么

父进程调用 `waitpid()` 时，拿到的不是“单纯退出码”，而是一个**编码后的进程状态**。

这个状态里通常会包含：

- 子进程是不是正常退出
- 如果正常退出，退出码是多少
- 子进程是不是被信号终止
- 如果被信号终止，是哪个信号
- 有时还会带一些额外标记，比如是否产生 core dump

所以从操作系统角度看：

- 父进程拿到的是“完整状态”
- shell 的 `$?` 是“压缩后的展示结果”

### 5.1 常见的解码方式

在 C 里通常会配合这些宏使用：

- `WIFEXITED(status)`
- `WEXITSTATUS(status)`
- `WIFSIGNALED(status)`
- `WTERMSIG(status)`

它们的意思分别是：

- 是否正常退出
- 正常退出码是多少
- 是否被信号杀死
- 被哪个信号杀死

## 6. `$?` 和 `waitpid()` 的关系

可以用一句话概括：

> `waitpid()` 给 shell 原始状态，shell 再把它转换成 `$?`。

如果子进程正常退出：

- shell 会取出退出码
- 然后把 `$?` 设成这个值

如果子进程被信号终止：

- shell 通常不会显示“原始 wait status”
- 而是转换成另一种更容易在脚本里使用的数字

## 7. 为什么退出码通常只有 0 到 255

这是 Unix 传统设计留下来的约定。

对于“正常退出”来说，最终能暴露给 shell 的退出码通常只有 8 位，也就是：

```text
0 ~ 255
```

所以如果程序源码里写：

```c
exit(812);
```

shell 最终通常只会看到低 8 位，也就是：

```text
812 % 256 = 44
```

这也是为什么一个程序“明明写了 812”，最后 `echo $?` 却看到 `44`。

## 8. 为什么信号终止常显示成 `128 + signal`

这不是内核强行规定给 shell 的原始值，而是 shell 常用的一种表示方式。

它的目的很简单：

- 正常退出时，`0~255` 已经被退出码占用了
- shell 又想把“被信号杀死”这件事编码成一个单独数字
- 所以很多 shell 约定用：

```text
128 + 信号编号
```

例如：

- `SIGKILL = 9`，常显示 `137`
- `SIGSEGV = 11`，常显示 `139`

这相当于 shell 在告诉你：

> 这个程序不是正常 `exit()` 结束的，而是被某个信号打死的。

## 9. 大白话理解“正常退出”和“信号终止”

可以把它想成两种离场方式：

- 正常退出：程序自己说“我干完了，这是我的结果”
- 信号终止：程序还没来得及自己交代，外部就把它打断了

所以：

- 正常退出时，程序自己提供退出码
- 信号终止时，shell 只能根据“死因”给你一个约定俗成的数字

## 10. 实际写程序时怎么用

### 10.1 对普通命令行程序

最常见约定是：

- `0` 表示成功
- 非 `0` 表示失败

### 10.2 在脚本里

shell 经常用 `$?` 或 `&&` / `||` 来做流程控制：

```bash
make && ./app
```

含义是：

- `make` 成功才继续执行 `./app`

### 10.3 在 C 程序里

如果你关心子进程怎么结束，通常应该用：

- `waitpid()`
- 配合 `WIFEXITED` / `WEXITSTATUS` / `WIFSIGNALED` / `WTERMSIG`

而不是只看一个简单整数。

## 11. 最后总结

最短记法：

- `return`：从函数返回
- `exit()`：做完用户态清理后结束进程
- `_exit()`：直接结束进程
- `$?`：shell 对上一个子进程结束状态的简化展示
- `waitpid()`：父进程拿原始结束状态的正式接口
- `128 + signal`：shell 对“被信号终止”的常见编码方式
